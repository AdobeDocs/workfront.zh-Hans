# 写入质量

使用Workfront文档样式指南和DQTI原则来审阅和改进Workfront文档。

## 工作流

查看或编辑内容时：

1. 读取目标文件
2. 应用以下规则 — 直接修复问题或标记问题
3. 保留作者的意图和技术准确性
4. 有关详细的术语、标点符号和程序规则，请参阅。claude/commands/_writing-quality-reference.md

## 语音和视点

- 使用&#x200B;**第二人**（“您可以……”） 为读取器寻址
- 用缩写来发声（别，不行，他们是）
- 使用“We recommendations”获得最佳实践，而不是“It recommendations”或“You should”
- 为一个用户组编写内容时，请引用第三者中的其他角色(“您可以查看……但是，Workfront管理员可以限制……”)
- 永远不要使用性别代词 — 重新构建句子或使用“他们”
- 没有用于选择的斜杠 — 使用“或”（“他/她”而不是“他/她”）

## 清晰度(DQTI)

- 每句一个想法
- 尽可能将句子保持在25字以下
- 以操作或结果开头，而不是后台上下文
- 使用最简单的词来传达含义（“使用”而不是“利用”、“开始”而不是“启动”、“关于”而不是“相关”）
- 避免进行提名（“创建”而不是“创建”、“配置”而不是“的配置”）
- 使用具体具体的语言 — 避免使用模糊的术语（“若干”、“各种”、“一些”）
- 避免双负值
- 使用可选代词(“that”、“who”)为句子结构增加清晰度

## 混凝度(DQTI)

- 使用特定示例而不是抽象描述
- 在示例中包含实际值
- 为确切的UI元素、字段和区域命名

## 任务方向(DQTI)

- 关注用户可以&#x200B;*做什么*，而不是系统&#x200B;*是什么*
- 具有命令性动词的潜在客户过程（“创建任务”、“配置通知”）
- 为用户提供足够的上下文以采取行动，但仅此而已
- 包含指向详细帮助的链接（“有关详细信息，请参阅[文章]。”）

## 大写

- **标题**：句子大小写始终（“创建任务”而不是“创建任务”）
- **Workfront术语**：仅在直接引用UI元素时大写
   - 直接引用：“完成‘规划完成日期’字段。”
   - 间接引用：“每个任务都必须有一个计划的完成日期。”
   - 提示：如果您能将“the”或“a”放在术语的前面，则通常应小写
- **按钮**：遵循用户界面大写，但全大写按钮→句子大小写除外
- **角色**： UI中角色的“系统管理员”；人员的“Workfront管理员”

## 标题

- 在句子大小写中使用命令命令（“创建任务”、“配置访问级别”）
- 标题应基于任务 — 描述用户将完成的任务
- 概述文章以“概述”（“项目概述”）结尾
- 使用清除的子目标将较长的部分分成多个标题

## 交叉引用和链接

使用这些精确模式：

| 状况 | 格式化 |
|-----------|--------|
| 提供信息后链接 | “有关详细信息，请参阅[文章名称]。” |
| 无先前信息的链接 | “有关信息，请参阅[文章名称]。” |
| 链接到另一篇文章中的章节 | “有关详细信息，请参阅[文章名称]中的[章节名称]。” |
| 链接到同一文章中的章节 | “有关详细信息，请参阅本文中的[分区名称]。” |

- 在段落中：与文本内联
- 在过程步骤中：在步骤之后的新行上
- 多个链接：使用项目符号列表
- 对屏幕位置使用“右上角”/“左上角”

## 粗体和斜体

- **粗体**&#x200B;可点击操作和UI元素（仅限过程步骤中）
- 请勿在过程步骤之外使用粗体UI元素
- 请勿粗体显示对话框名称、页面名称、图标名称或步骤之外的菜单名称
- 对用户应键入的文本使用&#x200B;*斜体*（“在URL框中键入&#x200B;*my.workfront.com*”）
- 绝不要使用斜体来强调 — 请改动句子

## 注释、提示和警告

谨慎使用 — 过度使用会让读者看不见它们。

- 每节最多一个注释/提示/警告
- 从不栈叠多个注释
- 将相关备注合并为一个带有项目符号列表的备注
- 不要使用注释来宣布新功能 — 请改动文章文本

| 类型 | 用途 |
|------|---------|
| `>[!NOTE]` | 不符合段落的信息；避免挫折；版本控制/兼容性 |
| `>[!TIP]` | 让用户的生活更轻松的善意建议 |
| `>[!IMPORTANT]` | 步骤或过程的重要信息 |
| `>[!WARNING]` | 提醒用户可能发生的数据丢失 |

## 程序

- 使用命令指示标题（“创建任务”）
- 仅在必要时才介绍 — 标题应足够了
- 首选简介格式：不定短语+冒号（“创建临时密码：”）
- 单步过程仍使用编号列表
- 使用“type”或“select” — 避免“set”或“specify”等模糊的词语
- 条件步骤：“（条件）如果您是多个团队的成员，请选择……”
- 可选步骤：“（可选）要添加表情符号，请单击……”
- 描述系统响应在发生之前或紧随其后的情况
- 多种方法：首先记录最直接的方法，然后使用“或”

### 程序中的右尖括号

- 使用`>`显示菜单/选项卡导航：“单击&#x200B;**电子邮件** > **通知**”
- 以粗体显示按钮名称，而不是括号
- 在括号中包含空格
- 请勿在主菜单导航中使用括号 — 请使用主菜单片段

## 标点快速参考

| 规则 | 示例 |
|------|---------|
| 始终使用牛津逗号 | “动机、手段和机会” |
| 在过程中，在“then”之前使用逗号 | “单击‘设置’，然后单击‘接口’。” |
| 日期中句子中的年后的逗号 | 2016年1月2日，第…… |
| 无仅包含月年的逗号 | “2016年1月” |
| 列表前冒号，列表后小写 | “选择下列选项之一：选项A” |
| 过程标题后无冒号 | “创建任务”（而不是“创建任务：”） |
| 短划线谨慎划线 | 如有可能，请重新构建句子 |
| 文件扩展名中的句点 | “上传.pdf文件” |

有关完整的标点符号和术语规则，请参阅。claude/commands/_writing-quality-reference.md。

## 完整性(DQTI)

- 包括用户需要理解和采取行动的所有信息
- 不要包含用户不需要的信息
- 始终提供指向详细文档的“更多信息”链接
- 文档系统默认值：列表中的“（默认值）”，或句子中的描述

## 组织(DQTI)

- 具有渐进式披露的逻辑结构（概述→详细信息→参考）
- 首先显示时间排序页面的最新内容（例如，发行说明）
- 每节一个主题
- 对3个以上的并行项目使用列表
- 使用表格进行结构化比较或字段说明
- 避免使用文本墙段落 — 分成可理解的块

## 文章类型

| 类型 | 用途 | 标题约定 |
|------|---------|-----------------|
| 概述 | 上下文、图表、工作方式 — 无过程 | 以“概述”结尾 |
| 操作说明 | 具有明确步骤的特定任务 | 命令动词 |
| 快速入门 | 新用户的设置信息+链接 | “开始使用……” |
| 参考 | 表格中的字段描述 | 描述性名词短语 |

- [ ]仅对过程步骤中的可点击操作使用粗体
- [ ]交叉引用使用标准格式（“有关详细信息，请参阅……”）
- [ ]个未栈叠的注释/提示/警告，每个部分最多一个
- [ 始终使用]牛津逗号
- [ ]过程步骤使用特定的动词（类型、选择、单击 — 不是“设置”或“指定”）

- [ ]没有性别代词
- [ ]句子清晰、具体且面向任务
