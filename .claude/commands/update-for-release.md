# 发布更新(Workfront)

此技能将逐步指导您为即将发布的功能更新Workfront帮助文章。 该工作流与`remove-preview-highlighting`相反：正在向文章添加新行为，这些行为被标记为“预览”，并（后来，在GA时）被该技能清除。

## 范围

当&#x200B;**all**&#x200B;为true时应用：

1. 用户正在更新发运功能的Workfront帮助文章（通常先预览）。
2. 此更改引入了新行为或UI，而不是GA清理。 对于GA清理，请使用&#x200B;**remove-preview-highlighting**。
3. 文件&#x200B;**不是**&#x200B;发行说明。 对于发行说明，请改用&#x200B;**发行说明格式化程序**。
4. 用户提供了功能上下文：至少是一个简短描述和一个屏幕截图；最好是PRD URL (Adobe Wiki)。

如果范围不明确，请先确认，然后再开始。

## 所需的工作流（循环中的人工）

请&#x200B;**不**&#x200B;批量编辑存储库。 一次移动一篇文章。 每篇文章后，询问是否继续阅读下一篇文章。

### &#x200B;1. 收集特征上下文

与用户确认：

- **更改的内容**（新行为或UI的1-2句摘要）。
- 新UI的&#x200B;**屏幕快照**。 如果提供，请在目标文章的`assets/`文件夹下使用描述性kebab大小写文件名保存（例如`add-custom-message.png`）。 如果未提供，请询问是要等待一个占位符引用，还是继续引用占位符。
- **PRD URL** (Adobe Wiki)（如果可用）。 使用`user-Adobe Wiki Confluence` MCP工具`get_wiki_content`获取它。 阅读本指南以了解用户在UI中看不到哪些行为：通知副作用、稍后编辑或添加某些内容时会发生什么情况、未显示字符限制、权限等。
- **可用性**：仅预览、预览+快速发布或已正式上市。 这促使在步骤3中选择代码片段。
- **显式排除项**：用户希望跳过的任何文章（例如“模板中没有此功能”）。

### &#x200B;2. 清点受影响的文章

使用功能区域的关键字（例如`approval workflow`、`document approval`、特定字段标签）搜索存储库。 构建候选列表：

- 相关`help/quicksilver/.../`树中的操作方法文章。
- 提及功能区域的概述文章和常见问题解答。
- **排除** `product-announcements/`（发行说明使用其他技能）。
- **排除**&#x200B;目录/索引页，其中唯一提及的是链接到另一篇文章的文本。
- **排除**&#x200B;用户在步骤1中说要跳过的文章。

向用户展示候选列表。 询问要更新的和跳过的。 如果同级文章似乎缺失，则交叉引用`help/quicksilver/TOC.md`。

### &#x200B;3. 选取预览代码片段

读取`help/_includes/snippets.md`并按可用性选择：

| 可用性 | 代码片段 |
|---|---|
| 仅预览 — 高亮显示的内容是GA文章中的新增内容 | `{{highlighted-preview}}` |
| 仅预览 — 整篇文章是新的 | `{{highlighted-preview-article-level}}` |
| 预览+快速发布客户，常规 | `{{preview-fast-release-general}}` |

如果当前季度已存在特定于版本的代码片段，请优先使用该代码片段，而不是通用代码片段。 在应用之前与用户确认选择。

### &#x200B;4. 每篇文章 — 首先显示，确定后编辑

对于用户确认列表中的每篇文章：

1. **读取文件。**

2. **确定突出显示模式。** 询问适合本文的用户（每篇文章的答案可能不同）：

   - **每个节重复**：将`in Production`附加到现有节标题。 添加附加了`in Preview`且包装在`<div class="preview"> ... </div>`中的新分区。 当新行为对过程有意义地更改时使用 — 额外的步骤、新图像、新表行或不同的措辞。 操作程序的典型用法。
   - **每行换行**：在现有部分中添加内联的新句子，在`<span class="preview"> ... </span>`中换行。 当添加的句子或两个自然适合现有段落、表格单元格或常见问题解答答案的句子时使用。
   - **混合**：同一文章中的某些部分使用每部分重复，其他部分使用每行包装。 当文章同时包含过程部分和FAQ样式部分时，可显示此选项。

3. **将代码片段**&#x200B;紧跟在H1标题的后面，并在上下加上空白行。 该代码片段位于&#x200B;**前面**&#x200B;介绍段落。

4. **将新详细信息存储为“始终包含”与“正在审核”。** 这是最重要的步骤。

   - **始终包含**（自动应用，无提示）：用户与UI交互时无法观察不可见的行为。 示例：
      - 副作用（例如，“编辑此项会将电子邮件重新发送给所有参与者”）
      - 在其他对象或以后事件中的行为
      - 先决条件和权限
      - UI中未显示限制
      - 用户只能从PRD、文档或产品团队学习的任何内容
   - **向上审阅**（以多选形式呈现给具有`AskQuestion`的用户）：用户使用该功能时可在屏幕上看到的事实。 示例：
      - UI已显示的字符计数器（例如`0 / 500`）
      - 字段的默认展开/折叠状态
      - 可见复选框的默认选定状态
      - 字段旁边的按钮标签
      - 内联显示的验证消息

   对于每个“等待审阅”项目，提供一句理由（“帮助新手计划更长的消息”，“帮助在以后阶段未看到消息的用户知道要扩展消息”）。 仅包括用户选取的项目。 默认原则是“如果用户在执行任务时可以在屏幕上看到它，请不要重述” — 但用户获得最终调用。

5. **建议编辑。** 在文章的摘录之前/之后显示（或重点突出的差异样式描述），包括：代码片段放置、标题重命名、新的预览中内容及其放置位置、屏幕快照引用以及任何内联`class="preview"`包装。

6. **等待明确批准**（“确定”、“申请”、“是”），然后再写入文件。

7. **验证。** 编写之后，对文件运行`ReadLints`并报告任何问题。 重新读取已更改的部分以确认结构。

8. **将文章级编辑**&#x200B;推迟到&#x200B;**写作质量**&#x200B;技能。 请勿在此重做语音、大写、粗体规则或链接模式 — 如果请求通过散文，请阅读`~/.cursor/skills/writing-quality/SKILL.md`。

### &#x200B;5. 每篇文章之后

询问是移至下一篇文章，还是停止、跳过或重新访问当前文章。

## 内容规则

### 标题

- 恰好将&#x200B;**`in Production`**&#x200B;附加到保留为生产端引用的现有节标题。
- 刚好将&#x200B;**`in Preview`**&#x200B;附加到新节标题。
- 将标题的其余部分保持句子大小写（每`writing-quality`）。

### 预览包装器

- **节级别**：在`<div class="preview"> ... </div>`中换行。 将开始标签和结束标签分别放在自己的行中，并在每个标签的上方和下方添加空白行，这样仍可呈现标签内的标题和列表。
- **内联（句子级别）**：在现有段落、表单元格或常见问题解答答案中以`<span class="preview"> ... </span>`换行。
- 从不在`<div class="preview">`中嵌套`<span class="preview">`。

### 代码片段放置

- 代码片段行紧跟在H1之后，上下均带有空白行。
- 代码片段位于&#x200B;**之前**&#x200B;介绍段落、`>[!IMPORTANT]`标注和任何访问要求块中。
- 每篇文章包含一个代码片段。

### 屏幕截图

- 使用描述性的kebab-case文件名将新屏幕截图保存到文章的`assets/`文件夹。
- 从新的“预览”部分中引用新的屏幕快照。 如果生产内部分的屏幕截图不再准确地反映功能，请将其保留在适当位置 — 在GA之前，它仍代表生产行为。
- 请勿虚构屏幕快照文件名；如果尚未提供任何屏幕快照，请询问用户。

### 注释和提示

- 每个节最多一个`>[!NOTE]`（或`>[!TIP]`、`>[!IMPORTANT]`、`>[!WARNING]`）。 如果现有部分已有注释，请将相关的新内容作为项目符号列表合并到同一注释中，而不是进行栈叠。

### 不该做什么

- 不编辑`product-announcements/`下的文章。
- 请勿批量编辑；每次只编辑一篇文章，每次都要获得明确批准。
- 不要在未向用户显示可观察的UI事实之前就将其加入。
- 除非用户明确指示，否则不要修改`<!-- ... -->`个HTML注释中的内容。
- 请勿更改`author:`或不相关的frontmatter字段。

## 呈现编辑内容之前的质量检查

- 代码片段在H1之后单独显示一次，上下均显示空白行。
- 现有节标题以`in Production`结尾。
- 新节标题以`in Preview`结尾，且节位于`<div class="preview">`内。
- 内联添加项位于`<span class="preview">`内。
- 已编辑文件上的`ReadLints`是干净的。
- 在这两种状态下，文章可正确阅读（预览内容显示并隐藏）。
