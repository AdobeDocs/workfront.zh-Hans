# 组和审批流程更改如何影响分配的审批流程

本文说明了当审批流程已与任务、问题、项目、模板或模板任务关联，并且Workfront管理员（或对审批流程具有管理访问权限的用户）执行以下操作之一时，会发生什么情况：

* 将审批流程（组级别）从一个组更改为另一个组
* 更改与项目关联的组
* 将审批流程从组级别更改为系统级别
* 将审批流程从系统级别更改为组级别

本文还介绍在来自不同组的两个项目之间移动或复制与组级审批流程相关的任务或问题时发生的情况。

有关可在Workfront中使用的3种审批流程类型的信息，请参阅[审批流程概述](../../../review-and-approve-work/manage-approvals/approval-process-in-workfront.md)。

>[!NOTE]
>
>如果项目或其任务或问题已使用组级自定义状态与组级审批流程关联，则更改组可能会导致上一个组的审批状态与系统级现有的审批状态之间发生冲突。
>
>例如，如果审批流程包含两个路径，一个用于系统级别状态，另一个用于等于相同系统级别状态的组级别状态，则更改原始项目的组将导致Workfront无法了解可能不存在于第二个组的特定于组的状态。 在这种情况下，您将遇到错误。
>
>在更新项目组之前，请考虑删除项目或其任务或问题的组级别审批流程。
>
>有关创建组级审批流程的信息，请参阅[组级审批流程](../../../administration-and-setup/manage-groups/work-with-group-objects/create-and-modify-groups-approval-processes.md)。
>
>有关创建组级自定义状态的信息，请参阅[创建或编辑组状态](../../../administration-and-setup/manage-groups/manage-group-statuses/create-or-edit-a-group-status.md)

## 将任务或问题移动或复制到组与审批流程不同的项目中

将任务或问题从项目移动或复制到另一个项目可能会影响任务或问题的现有审批流程，具体取决于两个项目的组。 下表说明了可能存在的方案：

| 原始任务或问题审批流程 | 两个项目的组 | 将任务或问题移至其他项目后审批流程中的更改 |
|---|---|---|
| 与系统范围状态关联的一次性审批流程 | 项目位于相同或不同的组中 | 无更改 |
| 与特定于组的状态关联的一次性审批流程 | 项目位于不同的组中 | 审批仍保持单次使用审批流程以及与审批更新关联的状态，以匹配新组的类似状态。 |
| 全系统审批流程 | 项目位于相同或不同的组中 | 无更改 |
| 组特定的批准流程 | 项目位于同一组中 | 无更改 |
| 组特定的批准流程 | 项目位于不同的组中，并且这些组具有不同的组特定状态 | 该审批会变成一次性审批流程，其状态与审批更新相关联，以匹配新组的类似状态。 |
| 组特定的批准流程 | 项目位于不同的组中，并且新组中的状态键与来自第一个组的审批流程关联的状态键相同 | 无更改 |
