# 创建新候选文章之前查找候选文章

当Courtney引入要合并的新内容时 — 引入文档、客户或PM反馈、Slack/转录粘贴、包含注释的屏幕截图、支持问题 — **不要建议先创建新文章。** 搜索存放库并显示可能适合其内容的候选现有文章。

## 所需的工作流

1. **首先阅读新内容**&#x200B;以识别主题、受众（管理员与最终用户）以及它添加的特定事实/步骤。
2. **使用grep搜索`help/`候选**&#x200B;并查找。 查找已涵盖相同功能、区域或工作流的文章。
3. **返回候选文章的排名列表**（通常为3-7），每个候选文章具有：
   - 文件路径（可单击的回勾路径）。
   - 它是候选者的一个一行原因（它会插入哪个部分，或者为什么它是最接近的主题匹配）。
   - 任何注意事项（例如，“受众不匹配 — 这是最终用户，引入是面向管理员的”）。
4. **在起草或编辑之前，询问Courtney放在何处**。 明确地用这个问来形容，例如，“我应该把它们写成哪一个？” 或者“希望我把它放入#2中，还是你认为它需要自己的文章？”
5. **仅在没有现有文章适合时建议新文章**，并解释为什么没有候选者有效。

即使内容“明显”属于已知文章，仍会显示2-3个替代项，以便Courtney能够确认投放位置。 这会捕获近乎重复的内容，以及内容已被部分覆盖的位置。

# 在编辑帮助文章之前提出更改

当将要编辑`help/`下的任何`.md`帮助文章时，**尚未编辑该文件**。 首先，显示您计划更改的内容并等待明确批准。

对于每个要触摸的文件：

1. 命名文件（路径）。
2. 将提议的更改显示为带防护的差异/片段（旧文本和新文本），周围有足够清晰的内容。
3. 简要说明理由（1句）。
4. 以明确的问题结束，例如“应用这些更改？” 或“要我继续吗？”

只有在Courtney同意后（或“执行”、“申请”、“执行”等） 是否应该调用编辑工具。

这适用于帮助`.md`文章上的&#x200B;**每**&#x200B;次编辑 — 拼写错误、链接修复、单字交换、术语清理、新部分和重写。 除非Courtney同时明确指出“只管修复它”或“不要询问，只管编辑”，否则不会有任何例外。

对于跨多篇文章的更改：将建议差异按文件分组到一个回复中。 如果文件集很大，请首先列出受影响的文件，每个文件的概要为1行，然后分批显示差异，并在每个批次之前进行确认。

这会&#x200B;**不**&#x200B;阻止只读研究(grep、read)或在聊天中起草/探索（无文件写入）。

# Git提交消息

在起草或生成Git提交消息时，请遵循此格式。

## 主题行

- 大约&#x200B;**50个字符或更少**。
- 总结&#x200B;**提示语气**&#x200B;中的更改（例如“添加……”、“修复……”、“重构……”）。

## 正文

- 在主文前面的主题后面留一行空白。
- 以大约&#x200B;**72个字符**&#x200B;换行。
- 请使用&#x200B;**项目符号行**&#x200B;来解释。 每个项目符号必须刚好以以下项之一开头：
   - **📖** — 在更改&#x200B;**添加**&#x200B;新内容时使用：新文件、新分区、新功能、新标头、新行或其他新字段内容。
   - **✏️** — 在更改&#x200B;**修改**&#x200B;现有工作时使用：编辑现有行、更新现有节、重构或更改当前内容。

示例：

```
Add refresh token rotation to auth flow

- 📖 Add refresh_tokens table and Alembic migration for schema v3.
- ✏️ Update session middleware to rotate secrets and revoke old tokens.
```

# 拉取请求(PR)

## 得出Jira问题

- 从&#x200B;**当前Git分支名称**&#x200B;中派生Jira问题ID（例如，与`FFENT-8796`匹配的区段）。
- **如果分支名称包含Jira ID：**，请在PR标题中使用该ID并将其链接到`# Context`→`## Jira`。
- **如果分支名称不包含Jira ID：**&#x200B;请忽略PR标题中的Jira密钥。 仍包括精确为`No ticket`的`## Jira`。

## PR标题

Jira ID为&#x200B;**的** `{type}/{JIRA-ID}- {short task description}`
示例： `feat/FFENT-8001- Add validation for numVariations in OCAPI request`

**无Jira ID：** `{type}- {short task description}`
示例： `docs- Refresh Object Composite API changelog`

### 提交类型

- **特征** — 添加了新功能
- **修复** — 修复错误
- **重构** — 重写/重构代码而不更改行为
- **性能** — 提高性能
- **样式** — 仅格式化/空白；无变化
- **测试** — 添加或更正测试
- **文档** — 仅文档，已添加新内容
- **生成** — 生成工具、CI、依赖关系、项目版本
- **操作** — 基础架构、部署、备份、恢复
- **杂项** — 其他（如`.gitignore`）

## PR正文 — 必需部分

### `# Summary`
简要概述更改了哪些内容以及更改原因。

### `# Changes`
按文件组织。 对于每个接触的文件，使用带有回勾号路径的2级标题，然后使用项目符号。

```markdown
# Changes

## `path/to/file.ext`
* Concise bullet describing the change in that file.
```

### `# Context`
始终包含`## Jira`子节。

```markdown
# Context

## Jira
[FFENT-8796](https://jira.corp.adobe.com/browse/FFENT-8796)
```

或者如果没有票证： `No ticket`
